# deploy/

Everything needed to deploy the Contract Engine stack to a Kubernetes cluster for testing.

## What gets deployed

Three services land in the cluster:

| Service | Chart | Purpose |
|---|---|---|
| **ds-contract-generator** | `server/contract-generator/charts/server` | Mints VC-JWT contracts |
| **ds-contract-validator** | `server/contract-validator/charts/server` | Verifies contracts |
| **ds-clearing-house-stub** | `server/clearing-house-stub/charts/server` | In-memory registry stand-in for the real Clearing House |

The stub Clearing House is **for testing only** — it's removed and replaced when the real Clearing House service exists. Contract Engine code does not change at that point; only the URL it points at.

## What you need before deploying

| Need | How |
|---|---|
| Container registry the cluster can pull from | Existing platform registry — same one used by `ds-catalog` / `ds-policy-engine` |
| `kubectl` context pointing at the target cluster | `kubectl config use-context <ctx>` |
| `helm` v3 | `helm version` |
| Ed25519 signing key for this node | Generated by [`scripts/generate-signing-key.sh`](scripts/generate-signing-key.sh) |
| Namespace to deploy into | Convention: same namespace as the rest of the node's Data Space services |

## Quick deploy — single node, all three services

```bash
# 1. Build and push images to your registry
export REGISTRY=registry.nextgen.hiro-develop.nl
export TAG=0.1.0

docker build -t $REGISTRY/ds-contract-generator:$TAG ./server/contract-generator
docker build -t $REGISTRY/ds-contract-validator:$TAG ./server/contract-validator
docker build -t $REGISTRY/ds-clearing-house-stub:$TAG ./server/clearing-house-stub

docker push $REGISTRY/ds-contract-generator:$TAG
docker push $REGISTRY/ds-contract-validator:$TAG
docker push $REGISTRY/ds-clearing-house-stub:$TAG

# 2. Generate this node's Ed25519 signing key and create the Secret
./deploy/scripts/generate-signing-key.sh nextgen hus.nextgen.hiro-develop.nl

# 3. Deploy all three services
./deploy/scripts/deploy-all.sh nextgen hus $REGISTRY $TAG
```

That runs the equivalent of:

```bash
helm upgrade --install ds-clearing-house-stub ./server/clearing-house-stub/charts/server \
  -n nextgen --create-namespace \
  -f deploy/values/hus-clearing-house.yaml \
  --set image.repository=$REGISTRY/ds-clearing-house-stub \
  --set image.tag=$TAG

helm upgrade --install ds-contract-generator ./server/contract-generator/charts/server \
  -n nextgen \
  -f deploy/values/hus-generator.yaml \
  --set image.repository=$REGISTRY/ds-contract-generator \
  --set image.tag=$TAG

helm upgrade --install ds-contract-validator ./server/contract-validator/charts/server \
  -n nextgen \
  -f deploy/values/hus-validator.yaml \
  --set image.repository=$REGISTRY/ds-contract-validator \
  --set image.tag=$TAG
```

## Verify the deployment

```bash
kubectl -n nextgen get pods
# Expected:
#   ds-clearing-house-stub-xxxx       1/1 Running
#   ds-contract-generator-xxxx        1/1 Running
#   ds-contract-validator-xxxx        1/1 Running

# Hit health checks
kubectl -n nextgen exec deploy/ds-contract-generator -- \
    curl -s http://localhost:8082/health-check/

kubectl -n nextgen exec deploy/ds-contract-validator -- \
    curl -s http://localhost:8083/health-check/
```

## End-to-end smoke test against the deployed services

Port-forward both services to your laptop and run the example script:

```bash
kubectl -n nextgen port-forward svc/ds-contract-generator 8082:8082 &
kubectl -n nextgen port-forward svc/ds-contract-validator 8083:8083 &

./examples/end-to-end.sh
# Expected: { "allow": true, ... }
```

## What's enabled / disabled by default in the charts

| Feature | Default | Where to flip |
|---|---|---|
| Generator's signing-key Secret mount | **enabled** | `signingKeySecret.enabled: true` in generator values |
| ServiceMonitor (Prometheus operator) | **disabled** | `serviceMonitor.enabled: true` in any chart |
| Ingress | **disabled** | `ingress.enabled: true` in any chart |
| HPA | not added — out of scope until traffic profile is measured |
| NetworkPolicy | not added — needs infra-team input on cluster CNI |
| Grafana dashboard ConfigMap | not added — waiting on infra team |

## Multi-node testing

To test cross-node validation, deploy the same three services to a second namespace with a different `nodeId`. Set the Validator's `jwksBaseUrlTemplate` to empty (`""`) so it derives JWKS URLs from the token's `iss` claim — then each node's Validator will fetch JWKS from the other node's Generator over the network.

Both nodes' Generator services need a `Service` of `type: NodePort`, `type: LoadBalancer`, or an `Ingress` so the cross-node JWKS fetch resolves. Set `ingress.enabled: true` in the Generator chart for that.

## Uninstalling

```bash
helm -n nextgen uninstall ds-contract-validator ds-contract-generator ds-clearing-house-stub
kubectl -n nextgen delete secret ds-contract-generator-signing-key
```
